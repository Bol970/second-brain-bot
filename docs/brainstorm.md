# Продуктовый брейншторм

## Главное обещание

Бот должен ощущаться как почти незаметный inbox для памяти. Главное правило: если отправка чего-то занимает больше нескольких секунд, система слишком тяжёлая.

## Режимы сохранения

- Обычный текст: сохранить как `thought`.
- Фразы намерения: «запиши мысль», «сохрани», «добавь» удаляются из начала сообщения, а полезный текст сохраняется.
- Идеи «что посмотреть»: «интересный фильм», `/movie`, «запиши что посмотреть» сохраняются как `movie`.
- Рецепты: `/recipe` или слова, похожие на рецепт, сохраняются как `recipe`.
- Ссылки: Tavily извлекает содержимое страницы; D1 хранит URL, краткое описание, теги, chunks и длинный raw content.
- Фото: R2 хранит медиа; D1 хранит подпись, теги, размеры, размер файла и ключи R2.
- Вопросы: «что посмотреть сегодня вечером», «найди рецепт...» ищут по базе и отвечают на основе сохранённых записей.
- Явный интернет-поиск: «поищи в интернете...» или `/web` используют Tavily Search, не смешивая результаты из интернета с ответами по личной базе без явной команды.
- Задачи/напоминания: `/task`, `/remind`, `/reminders` и `/done` держат последующие действия в том же боте.

## Решения по хранению

- D1 - основной источник правды для метаданных, тегов, поискового текста, chunks и состояния записей.
- R2 - основной источник правды для бинарных медиафайлов.
- Длинные веб-страницы по умолчанию сохраняют raw extracts до защитного лимита Worker/D1 и режутся на chunks для поиска.
- `/compact id` удаляет raw extract только по запросу, оставляя summary, chunks и поисковые поля.
- Теги нормализуются в отдельную таблицу, чтобы потом было проще делать UI и фильтрацию.
- Взаимодействия логируются по минимуму — для отладки и, может быть, персонализации позже.

## Решения по поиску

Начинаем с гибридного лексического поиска в D1:

- вывести вероятные типы записей из запроса;
- разбить запрос на короткие ограниченные токены;
- искать по `search_text` и тегам;
- переупорядочить результаты по title, summary, tags, body, importance и совпадению type;
- использовать OpenRouter, чтобы собрать ответ из найденных записей.

Будущее улучшение:

- добавить Cloudflare Vectorize и embedding model, когда сохранённый архив вырастет за пределы того, что хорошо покрывает лексический поиск и поиск по тегам.

## Зафиксированное поведение

- Локальные вопросы ищут только по сохранённой базе.
- Если локальных совпадений нет, бот предлагает интернет-поиск вместо автоматического ухода в интернет.
- Неоднозначное сохранение может вызвать уточняющий вопрос.
- Длинные raw extracts хранятся до ручной команды compact.

## Решения по медиа

Cloudflare Workers не умеют запускать тяжёлые нативные библиотеки для изображений вроде Sharp. Что работает в рамках бесплатного лимита:

- выбирать уже уменьшенный вариант фото из Telegram;
- ограничивать media размером 8 MB по умолчанию;
- пробовать трансформацию Cloudflare `cf.image` перед сохранением в R2;
- хранить в D1 только метаданные и ключи R2;
- следить за R2: сверх бесплатного месячного лимита он платный.

## Открытые вопросы

1. Нужен ли приватный веб-интерфейс для просмотра, редактирования, объединения тегов и удаления записей?
2. Нужно ли хранить изображения навсегда или стоит добавить автоматическую политику «оставлять только сжатую версию»?
3. Какой интервал для напоминаний лучше: 5 минут, 15 минут или каждый час?
